# Runbook - Phase 0.4: Keycloak (Identity & Access Management)

This runbook covers deploying **Keycloak** - the single identity provider for the platform -
backed by its **own private MSSQL** (Database-per-Service), exposed to the outside world only
through the **YARP gateway** (single public entry point). Keycloak mints every JWT; it is
validated at YARP and re-validated at each service (defense in depth, wired up from 0.4 onward).

> Prerequisites: Phases 0.1-0.3 healthy (cluster, Linkerd CNI, YARP gateway with a MetalLB IP),
> and a container builder (`docker`/`nerdctl`) on the VM (the gateway image is rebuilt with the
> new Keycloak routes).

## Scope of 0.4 (dev)

Implemented now:

- **Private MSSQL** for Keycloak (`keycloak-mssql` StatefulSet, Longhorn PVC, mTLS via Linkerd
  with `opaque-ports: 1433`). No other service may touch this database.
- **DB bootstrap Job** that creates the `keycloak` database + a least-privilege `keycloak` login
  (`db_owner` on that DB only). Keycloak manages its own schema thereafter.
- **Keycloak 26.6.3** in production `start` mode, `KC_DB=mssql`, behind the proxy
  (`KC_PROXY_HEADERS=xforwarded`, TLS terminated at the gateway), health on the management port.
- **Realm import** (`itorchestra-dev`): `api-gateway` confidential client (Auth Code + PKCE +
  service account, audience + roles mappers), realm roles `platform.admin`/`platform.user`, and a
  dev test user.
- **YARP routes** for `/realms`, `/resources`, `/admin`, `/js`, `/robots.txt` -> Keycloak.

Deferred: enabling **JWT validation** in YARP and the services (next), MFA/lockout hardening,
and audit-log export to OpenSearch.

## Decisions (this environment)

- **Database = MSSQL** (consistent with the platform's MSSQL-everywhere rule). Keycloak owns a
  private instance; cross-service DB access is forbidden.
- **Exposure = via YARP** only. Keycloak has no LoadBalancer; `KC_HOSTNAME` is set to the
  gateway's public `https://<ip>` so issuer/redirect URLs are correct behind the proxy.
- **Realm imported** declaratively on startup (`--import-realm`).

## Deploy (dev)

```bash
cd ~/itOrchestra/platform
export KUBECONFIG="$HOME/.kube/config"

# one-shot: rebuild gateway (new routes) -> roll it -> deploy Keycloak+MSSQL -> verify
bash bootstrap/03-keycloak-dev.sh
```

Or step by step:

```bash
# 1) gateway now carries the Keycloak routes -> rebuild + reimport + roll
bash gateway/build-and-import-dev.sh
kubectl -n ns-gateway rollout restart deploy/gateway
kubectl -n ns-gateway rollout status  deploy/gateway --timeout=180s

# 2) Keycloak + private MSSQL (secrets, MSSQL, DB-init Job, realm, deployment, NetworkPolicy)
bash k8s/keycloak/install-dev.sh

# 3) verify
bash bootstrap/verify-0.4.sh
```

## Verification

`bootstrap/verify-0.4.sh` checks:

- `keycloak-mssql-0` is `Ready`.
- The `keycloak-db-init` Job succeeded.
- The Keycloak pod is `Ready` and meshed (`linkerd-proxy` sidecar).
- `GET https://<gw-ip>/realms/itorchestra-dev/.well-known/openid-configuration` -> `200` and the
  `issuer` equals `https://<gw-ip>/realms/itorchestra-dev`.
- The JWKS endpoint returns signing `keys`.

Useful ad-hoc commands:

```bash
IP=$(kubectl -n ns-gateway get svc gateway -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

# OIDC discovery + JWKS (self-signed gateway cert -> curl -k)
curl -sk https://$IP/realms/itorchestra-dev/.well-known/openid-configuration | jq .
curl -sk https://$IP/realms/itorchestra-dev/protocol/openid-connect/certs   | jq .

# Get a dev token (password grant; dev only) for the test user
curl -sk -X POST https://$IP/realms/itorchestra-dev/protocol/openid-connect/token \
  -d grant_type=password -d client_id=api-gateway \
  -d client_secret=dev-gateway-secret-change-me \
  -d username=dev -d password=devpass | jq -r .access_token

# Admin console (browser): https://$IP/admin/  (user: admin)
kubectl -n keycloak get secret keycloak-admin -o jsonpath='{.data.password}' | base64 -d; echo
```

## Configuration

- **Image**: `quay.io/keycloak/keycloak:26.6.3` (pinned).
- **DB URL**: `jdbc:sqlserver://keycloak-mssql.keycloak.svc.cluster.local:1433;databaseName=keycloak;encrypt=true;trustServerCertificate=true`.
- **Secrets** (generated by `install-dev.sh`, never in Git): `keycloak-db` (`sa-password`,
  `kc-password`) and `keycloak-admin` (`username`, `password`). In prod these move to Vault (0.5).
- **Realm**: `k8s/keycloak/realm-configmap.yaml`. The `api-gateway` secret + dev user password are
  DEV placeholders - rotate/remove for any non-dev environment.
- **Token TTL**: access token 15 min; refresh-token rotation on; brute-force protection on.

## Troubleshooting

| Symptom | Cause / fix |
|---|---|
| MSSQL `CrashLoopBackOff` / OOM | SQL Server needs >= 2 GiB; check the VM has headroom (`kubectl -n keycloak describe pod keycloak-mssql-0`). |
| MSSQL pod not Ready, proxy stalls | `opaque-ports: "1433"` annotation missing on the pod/service (TDS is server-speaks-first). |
| `keycloak-db-init` Job fails | SA password complexity or MSSQL not up yet; check `kubectl -n keycloak logs job/keycloak-db-init`. |
| Keycloak `start` fails on hostname | `KC_HOSTNAME` not set / gateway IP missing; re-run `install-dev.sh` (it reads `svc/gateway`). |
| Discovery issuer is `http://...` or wrong host | gateway not forwarding `X-Forwarded-Proto`, or `KC_PROXY_HEADERS!=xforwarded`. |
| Admin console assets 429 | edge rate limit; IdP paths use the relaxed 300/min bucket - rebuild/roll the gateway if it predates the route change. |
| `/realms/...` 404 via gateway | gateway image predates the Keycloak routes - rebuild + `rollout restart deploy/gateway`. |
| Keycloak can't reach DB | wrong `kc-password` (secret regenerated after DB init) - delete the DB-init Job + secret consistently, or reset the login password. |

## Production notes

- HA: 2+ Keycloak replicas (cache `ispn`/jgroups), MSSQL with proper sizing/backup; or a managed
  SQL instance. Pin and pre-build the image (`kc.sh build` + `--optimized`).
- Real TLS end-to-end; `KC_HOSTNAME` = the real public FQDN. `KC_HOSTNAME_STRICT=true`.
- All secrets from Vault; no embedded realm secrets. Provision realm/clients via IaC, not a dev
  ConfigMap with inline credentials.
- Enable MFA for admins, account lockout, and forward auth audit events to OpenSearch.

## Teardown (dev)

```bash
kubectl -n keycloak delete deploy/keycloak svc/keycloak job/keycloak-db-init \
  configmap/keycloak-realm-itorchestra-dev 2>/dev/null
kubectl -n keycloak delete statefulset/keycloak-mssql svc/keycloak-mssql
kubectl -n keycloak delete pvc -l app=keycloak-mssql        # WARNING: deletes Keycloak's data
kubectl -n keycloak delete secret keycloak-db keycloak-admin
kubectl -n ns-gateway delete networkpolicy allow-egress-to-keycloak
```
